# oosync Architecture Guide

This is the top-level architecture guide for the oosync repository.

oosync is a schema-agnostic offline sync toolkit. It has three major responsibilities:

1. Introspect a Postgres schema and generate consumer-owned sync artifacts.
2. Provide a portable sync runtime that moves data between a local SQLite database and a remote worker.
3. Provide a generic worker runtime that applies sync policy against Postgres using injected artifacts.

The repository is intentionally consumer-agnostic. Application-specific behavior belongs in generator inputs, configuration, or consumer wiring, not in oosync runtime code.

## Repo Topology

- `src/codegen-schema.ts`
  - Postgres introspection and artifact generation.
- `src/shared/**`
  - Shared protocol types and portable metadata helpers.
- `src/runtime/**`
  - Platform/runtime abstractions used by the sync engine.
- `src/sync/**`
  - Core sync runtime: outbox handling, push/pull orchestration, adapters, remote-change application, realtime integration, and service lifecycle.
- `worker/src/index.ts`
  - Generic worker factory and request handling.
- `worker/src/sync-schema.ts`
  - Config-driven schema helpers for worker filtering, sanitization, and rule evaluation.
- `worker/src/debug.ts`
  - Worker diagnostics helpers.

## Mental Model

Use this model when reasoning about changes:

1. Postgres is the remote authoritative schema.
2. `src/codegen-schema.ts` inspects that schema and derives sync facts.
3. oosync writes generated outputs into consumer-owned paths.
4. The consumer app runs against local SQLite and queues local writes.
5. The oosync runtime pushes queued changes to a worker and pulls remote changes back.
6. The worker uses generated schema/config artifacts to apply sync logic against Postgres.

The live runtime source of truth for the consumer is local SQLite. oosync is the transport, normalization, and reconciliation layer around that database.

## Generated vs Hand-Written

Generated outputs are part of the contract, but they are not owned by this repository at runtime.

Generated by `src/codegen-schema.ts` into consumer-owned paths:

- SQLite Drizzle schema
- optional SQLite schema wrapper that adds the sync outbox table
- shared/generated table metadata
- consumer-facing table metadata projection
- worker Postgres schema
- worker config
- optional worker entrypoint scaffold
- optional SQLite migration support files when configured

Hand-written code that remains in this repository:

- the code generator
- shared protocol definitions
- runtime interfaces and portable helpers
- sync engine, outbox, adapters, and service lifecycle
- generic worker runtime and rule interpreter

Rule: do not hand-edit generated consumer outputs to fix behavior. Fix the generator, the Postgres schema, table comments, or config inputs instead.

## Artifact Flow

Artifact flow is how schema facts become runtime contract:

1. A consumer defines or changes remote Postgres schema.
2. The consumer runs `npm run codegen:schema` or invokes `src/codegen-schema.ts` directly.
3. oosync introspects catalogs, constraints, column types, foreign keys, indexes, and table comments.
4. oosync merges inferred facts with consumer overrides from `oosync.codegen.config.json`.
5. oosync writes generated artifacts into consumer-owned output paths.
6. The consumer app and consumer worker import those outputs.

The key architectural point is that schema agreement is generated, not maintained manually in multiple places.

## Runtime Flow

Runtime flow is how data moves after a user changes local state:

1. A consumer writes to local SQLite immediately.
2. Local triggers or queue logic append changes to the outbox.
3. `SyncService` and `SyncEngine` package pending changes and request remote updates.
4. The worker authenticates the request, applies push logic to Postgres, evaluates pull rules, and returns normalized changes.
5. The sync runtime applies remote changes back into local SQLite while suppressing local-trigger feedback loops.
6. The consumer continues to read local SQLite; it does not switch to direct remote reads after sync.

Default conflict handling is last-write-wins unless the consumer layers additional conflict behavior above the shared protocol.

## Key Boundaries

### 1. Codegen Boundary

- `src/codegen-schema.ts` is the source of truth for generated artifacts.
- Generated outputs are consumer-owned and write-only.
- Codegen should prefer inference from Postgres catalogs and comments before requiring manual overrides.

### 2. Shared Contract Boundary

- `src/shared/**` must remain portable and data-oriented.
- Protocol types should not import consumer code or worker-specific runtime state.
- Shared metadata is contract, not application logic.

### 3. Library Runtime Boundary

- `src/**` must remain independent of `worker/**`.
- The sync runtime depends on injected runtime/platform hooks rather than browser-framework or worker-specific globals.
- Consumer-specific notification, storage, auth wiring, and schema objects should enter through configuration or runtime injection.

### 4. Worker Boundary

- `worker/**` owns generic request handling, auth validation, Postgres access, push application, and pull evaluation.
- The worker must receive schema tables, syncable table names, metadata, and worker config from injected artifacts.
- Worker logic must not hard-code consumer schema assumptions.

### 5. Consumer Boundary

- Consumer-specific table names, ownership rules, collections, and domain semantics belong in config, generated artifacts, or consumer code.
- If a behavior is only correct for one application, it does not belong in oosync runtime code.

## Configuration Inputs

oosync primarily learns about a schema from these inputs, in this order:

1. Postgres catalogs and constraints
2. Postgres table comments with `@oosync.*` tags
3. `oosync.codegen.config.json`

Use overrides to express exceptions, not as the default modeling mechanism.

## Runtime Components

### Library Runtime

- `src/sync/engine.ts`
  - Push/pull orchestration, ordering, retry behavior, incremental sync state.
- `src/sync/outbox.ts`
  - Durable queue utilities and local-row lookup helpers.
- `src/sync/apply-remote-changes.ts`
  - Remote-row application into SQLite with conflict-target handling.
- `src/sync/service.ts`
  - Sync lifecycle, scheduling, realtime coordination, and consumer-facing control surface.
- `src/sync/realtime.ts`
  - Optional realtime integration that triggers pull work.
- `src/sync/runtime-context.ts`
  - Injected runtime dependencies such as local schema, queue table, sqlite instance access, persistence hooks, and logging.

### Worker Runtime

- `worker/src/index.ts`
  - `createWorker(...)`, auth verification, database connection setup, request processing, and sync response shaping.
- `worker/src/sync-schema.ts`
  - Table metadata helpers, pull-rule evaluation, push sanitization, and collection loading.
- `worker/src/debug.ts`
  - Diagnostic output control.

## Local Development

Primary repository validation commands:

- `npm run typecheck`
- `npm run lint`
- `npm run check`
- `npm run test`

Codegen DB selection prefers `OOSYNC_DATABASE_URL`. `DATABASE_URL` is only accepted automatically when it clearly targets a local Supabase-style database endpoint.

## Change Workflows

### Schema / Codegen Change

1. Adjust Postgres schema, comment tags, or expected inference behavior.
2. Update `src/codegen-schema.ts` if inference or output generation must change.
3. Regenerate artifacts in the consumer project.
4. Run the drift check path.
5. Validate that runtime code still matches the generated contract assumptions.

### Sync Runtime Change

1. Identify whether the change belongs in shared contract, runtime context, engine behavior, or worker behavior.
2. Preserve the separation between `src/**` and `worker/**`.
3. Validate the smallest relevant commands and tests.

## Guidance For AI Agents

When changing this repository, first decide which layer owns the work:

- Generator/input modeling issue: start with `src/codegen-schema.ts`.
- Shared protocol or metadata issue: start with `src/shared/**`.
- Local sync runtime issue: start with `src/sync/**` and `src/runtime/**`.
- Worker-side push/pull behavior issue: start with `worker/src/**`.

Do not solve consumer-specific problems by hard-coding consumer logic into oosync. The durable architecture is: schema and config as inputs, generated artifacts as contract, local SQLite as the consumer runtime source of truth, and oosync as the generic sync engine and worker implementation.
